# 07 — Implementation Rules

## Build rules

- Start from the documented identity; do not invent a new visual style during coding.
- Use real personal information and verified project details only.
- Keep content, projects, tools, and links in typed data files.
- Use CSS variables for colors, spacing, borders, and motion timings.
- Prefer CSS and Motion for most interactions.
- Introduce GSAP or a video only when the visual purpose is clear.

## Definition of done

The portfolio is ready when:

- The hero identifies El Houssaine and his Data Analyst specialization within seconds.
- The complete BI project is the strongest and clearest proof.
- The site feels personal, not like a generated template.
- Every animation has a reason and remains smooth on mobile.
- Contact, CV, GitHub, and LinkedIn are easy to find.
- No placeholder text, fake metrics, broken links, console errors, or overflow remain.
- The production build succeeds and the Vercel deployment works on a clean device.

## Final review questions

Before shipping, ask:

1. Could another Data Analyst use the same site unchanged?
2. Does every visual choice support clarity or personality?
3. Can a recruiter understand the strongest project without a long explanation?
4. Does the site still work when all animation is disabled?
5. Does the writing sound like Houssaine rather than a language model?
